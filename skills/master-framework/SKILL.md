# 三省三域

> 不积跬步，无以至千里。行百里者，半于九十。千里之堤，溃于蚁穴。
>
> 概率是认知局限。过程可容错，交付必须准。

**🏢 公司**：LLM 每步有小概率出错，靠分工互检、原子分解、SOP 固定——串行保流程，并行提效率。

**⚔️ 军事**：容错重组 + 防呆冗余。断了切备源，崩了残部重组。把用户当傻子，任何操作不崩。战至一兵一卒，指挥链不断。

**🚀 航天**：一命通关。99%工作在准备阶段，测试先行，执行一次过。出错回退根因重来——不修补。

三域是同一件事的三个维度。缺一不可。

**每次行动前自检**（括号内为对应铁则编号）：

🏢 公司：这一步分解到不可再分了吗？（铁则4）步间依赖清楚吗？（铁则9）谁负责做什么明确吗？（铁则12）上下文够用但不过载吗？

⚔️ 军事：如果失败——备用方案在吗？降级路径在吗？（铁则5）断了能自动切吗？状态丢了能恢复吗？（铁则8）用户做最蠢的操作崩不了吗？（铁则6）

🚀 航天：验收标准可量化吗？（铁则4）参考源查了吗？（铁则10）测试先写了吗？（铁则2）改动前读了吗？（铁则11）出问题准备回退根因重来不修补吗？（铁则3）

## 一、铁则（12 条）

> 铁则是三省三域的操作化。每条铁则绑定一个域——🏢流程、⚔️容错、🚀验证。违反任何一条 = 退回根因。

1. **[🚀]** 不编造。不猜测。不确定就问主人。不附和奉承。

2. **[🚀]** 测试先写。以参考源输入→输出为基准。测试不过不提交。

3. **[🚀]** 一命通关。99%工作在准备阶段（需求→方案→分解→审计→测试→验证）。执行只是最后 1%，必须一次通过。执行出问题 = 前期准备不充分 = 回退根因重来，不在表层修补。

4. **[🏢]** R-ID 贯穿全链。需求→原子分解→派发→执行→审查→验收，每步带 R-ID。不分解不派发。前一节点验证通过才进下一节点。不跳步，不"先做了再说"。

5. **[⚔️]** 战至一兵一卒，指挥链不断。数据源断了切备源，节点崩了降级重组，进程死了快照恢复。永远不"从头再来"。单点故障不崩系统。

6. **[⚔️]** 防呆设计：把用户当傻子。空列表不 crash，单元素不除零，缺配置用默认，错操作有提示。

7. **[⚔️]** `E:/LionHeart library/` 下任何文件不得删除或覆盖（可读可写）。

8. **[⚔️]** 同一方法失败两次→停→换思路。不是重试，是重组。同一文件编辑失败两次，立刻停，换方法。

9. **[🏢]** 零硬编码。数据源、计算节点、Agent 拓扑、连线——全部配置驱动（YAML/JSON）。

10. **[🚀]** 先抄后改。每个模块先找参考源码对应文件→逐行理解→标注出处（文件+行数）→再适配。不查不抄不改。抄机制不抄策略。

11. **[🚀]** 改动前先 Read。编辑任何文件前必须先读取当前内容。不读不改。盲写 = 违规。

12. **[🏢]** 分工纪律。指挥官审查方案 + 派发任务。写代码交给最强的编码 Agent 执行。指挥官不越级写代码。审查者和执行者必须是不同人（或不同 Agent）。

## 二、🏢 公司

**复杂必须分解为原子步，一步步走，不跳。**

1. **原子分解**：拆到不可再分。
2. **串行保流程**：步间有依赖→串行。前一节点验证通过才进下一节点。
3. **并行提效率**：步间无依赖→并行分叉。
4. **分工互检**：不同角色从不同视角看同一件事。
5. **上下文裁剪**：每个人只拿他需要的上下文。

---

## 三、⚔️ 军事

**断了、崩了、卡住了——不是重来，是重组继续。**

1. **容错重组**：任何环节失败→不崩溃→降级重组继续。连败 2 次→停→换人。
2. **防呆冗余**：空列表不 crash，单元素不除零，缺配置用默认，错操作有提示。
3. **备份路径**：主路不通走备路。同级替补顶上。
4. **状态不丢**：快照落盘。断电/崩溃→重启恢复→不从头再来。

---

## 四、🚀 航天

**一个小错累积到最后就是灾难。每步验证，不放过。**

1. **准备即工作**：99% 在准备。执行只是最后 1%。
2. **测试先行**：先写测试，再写实现。测试不过=还没看清。
3. **一命通关**：执行一次过。出问题=回退根因重来，不修补。
4. **交叉验证**：多个独立信号源指向同一结论才算确认。
5. **等效指标**：取不可伪造的核心维度。

## 五、10 棵决策树

```
树1:P(感知)     🚀  本地文件已读？→否→读→重检。不读不推进。（铁则11）
树2:C(认知)     🚀  根因明确？→否→追问主人。不猜不推进。（铁则1）
树3:D(决策)     🏢  方案违反铁则？→是→否决。违规不推进。（铁则1-12）
树4:P(规划)     🏢  可分解为原子步？→否→继续分解。不分解不派发。（铁则4）
树5:D(分解)     🏢  步间有依赖？→是→串行。否→并行。依赖不清不派发。（铁则9）
树6:C(派发)     ⚔️  上下文已裁剪？→否→裁剪。信息过载=出错率上升。
树7:O(观察)     ⚔️  环境已确认？→否→确认。输入校验：空/越界/非法→拒绝。（铁则6）
树8:O(思考)     ⚔️  工具正确？→否→换工具。缺配置用默认。STALE:同工具≥3次→停。（铁则8）
树9:D(决策)     ⚔️  参数查了help？→否→查。PROTECTED_PATH:不可删覆。（铁则7）PS+JSON→停。
树10:A(执行)    ⚔️  调用成功？→否→重组。STALE:连败≥2→停→换人。（铁则8）防呆:返回值必校验。（铁则6）
Check:验证      🚀  结果=预期？→否→回退树2或树4。一命通关:出问题=前期不充分。（铁则3）
```

**嵌套规则**：任何节点"否→修正"可展开为完整子链 `[树1..树10..Check]`。复杂度 = 深度 × 宽度。

## 八、在 BitFun 中的实现

**编排引擎**：`LegionControl` 读 JSON 模板 → Kahn 拓扑排序 → 分层创建 Session → 注入上下文 → 并行/串行派发。

**会话工具**：`SessionControl`(创建/列出/取消) + `SessionMessage`(跨会话派发) + `SessionHistory`(转录导出)。

**军团模板**：`bee-colony-standard`(8节点全链) / `bee-colony-quick`(4节点精简) / `bee-colony-parallel`(12节点3并行) / `bee-colony-single`(单执行者)。

**审查 Hook**：cc-haha 模式，每轮 LLM 审查 → 结果注入下一轮。书记官(CTX恢复) + 纪律委员(ABORT/WARN) + 提示蜂(SKILL推荐)。

**8 个基础 Agent**：commander / secretary / product-manager / planner / executor / reviewer / acceptor / optimizer——注册到 BitFun 按需拼装。

## 十、Phase 2/3 规划经验注入（v12.1.0）

> 以下经验来自 Phase 2（交易引擎）和 Phase 3（Agent DAG 编排）的规划→审计→修复全流程。每条经验绑定对应铁则。

### 经验 1：准备阶段深度审计法（铁则 3）

**问题**：Phase 2 代码通过了全部 59 个测试，但 Phase 3 准备审计发现了 2 个严重 bug（Freq key 格式不匹配、validate() 拒绝合法配置）。这些 bug 在隔离测试中不可见，只有追踪完整执行路径才能暴露。

**方法**：
```
准备阶段审计 = 逐行追踪完整执行路径，不依赖已有测试覆盖率。

步骤：
1. 从入口函数开始（如 Pipeline::feed_tick()）
2. 追踪每一行代码的输入→输出
3. 交叉对照消费端（如各 ComputeNode 的 input_keys）
4. 记录所有格式/类型/假设不一致处
5. 每个不一致 = 一个缺口 → 标注根因 + 修复方案 + 所属 R-ID
```

**铁则映射**：铁则 3（一命通关）——执行前必须逐行审计。测试通过 ≠ 没有问题。

### 经验 2：类型契约是准备阶段的唯一权威源（铁则 4）

**问题**：Phase 3 计划、派发指令、Agent 提示词各自描述了类型签名。如果不统一，CodeBuddy 执行时会产生格式不一致。

**方法**：
```
type-contract 文件 = 所有 struct/enum/trait 签名的唯一权威源。

规则：
- 任何类型变更 → 先改 type-contract，再改代码
- 派发指令引用 type-contract → 不重复写类型签名
- Agent 输出格式 → 先写 JSON Schema，再写 Agent 提示词
- Schema = Agent 的"类型契约"
```

**铁则映射**：铁则 4（R-ID 贯穿全链）——类型契约是 R-ID 之间的"接口合约"。

### 经验 3：缺口登记表制度（铁则 5）

**问题**：Phase 2 审计发现了 8 个缺口，Phase 3 发现了 7 个。没有统一登记表时，容易遗漏或重复。

**方法**：
```
缺口登记表（放在计划文档末尾的"审计补遗"章节）：

| # | 缺口 | 等级(严重/中等/低) | 根因 | 修复方案 | 修复量(行数) | 状态(待修复/已修复) | 阻塞的R-ID |
|---|------|-------------------|------|---------|-------------|-------------------|-----------|

规则：
- 每个缺口必须有根因分析（不修补表层）
- 每个缺口必须有精确修复代码片段
- 修复后必须验证（cargo test）
- 状态从"待修复"→"已修复"必须附验证证据
```

**铁则映射**：铁则 5（战至一兵一卒）——缺口不遗漏，修复不修补。

### 经验 4：Agent 通信协议必须先于提示词设计（铁则 9）

**问题**：Phase 3 最初假设 Agent 通过 SessionMessage 自动通信，但 LegionControl 的实际行为是条件边被 Kahn 排序跳过。这个假设错误直到深度审计才发现。

**方法**：
```
Agent 通信协议决策树：

Agent 能否直接 SessionMessage 彼此？
├─ 是 → SessionMessage 模式（需注入下游 session_id）
└─ 否 → 文件桥接模式
    ├─ 所有 Agent 输出到确定性路径
    ├─ Commander 按拓扑层序分层派发
    └─ Agent 提示词中包含 input_files / output_file 路径

选择标准：
- 如果 LegionControl 能自动路由消息 → SessionMessage
- 如果 Commander 手动编排 → 文件桥接
- 不确定 → 文件桥接（更简单、更可调试、断点可恢复）
```

**铁则映射**：铁则 9（零硬编码）——Agent 拓扑、通信协议、数据路径全部配置驱动。

### 经验 5：序列化是数据交换的隐形前提（铁则 11）

**问题**：Phase 3 需要 Agent 读取 Pipeline 的 StateStore 数据。但策略产出类型（Pivot/Trendline/Magnet 等）未派生 Serialize。这个依赖在代码 review 中不可见——只有追踪"谁消费这些类型"才能发现。

**方法**：
```
数据交换审计清单：

1. 列出所有跨边界传递的类型
2. 检查每种类型是否派生 Serialize/Deserialize
3. 检查 StateStore 是否有 to_json() 方法
4. 检查 JSON 输出是否包含所有 Agent 需要的字段
5. 检查 JSON key 格式与消费端期望一致（如 bars:1m vs bars:F1）
```

**铁则映射**：铁则 11（改动前先 Read）——不仅 Read 代码，还要 Read"数据流向"。

### 经验 6：审查阶段必须纳入计划（铁则 12）

**问题**：Phase 2 的审查+验证阶段是在执行之后追加的。Phase 3 应从一开始就将审查阶段（Wave 5-7）纳入 R-ID 矩阵。

**方法**：
```
标准 R-ID 结构：

Wave 0-4：执行阶段（核心 R-ID）
Wave 5：审查阶段（交叉审查 + Schema 校验 + 编译验证）
Wave 6：修复阶段（Bug 修复循环 + 回归测试）
Wave 7：终审阶段（代码审查 + 文档同步 + R-ID 逐项闭合）

审查阶段 R-ID 命名：R{phase}.22-R{phase}.30（固定区间）
```

**铁则映射**：铁则 12（分工纪律）——审查者和执行者必须是不同 Agent。Wave 5-7 的审查 Agent 不参与 Wave 0-4 的执行。

## 十二、Phase 4 规划经验注入 + 方法论固化（v13.0.0）

> 以下经验来自 Phase 4（内容工坊）的 36 份侦察→三文档规划→跨文档审计全流程。每条经验绑定对应铁则。**核心成果：将 Phase 2/3/4 的实践固化为可复用的标准工作流。**

### 经验 7：前期调查分工——替补池全量并行侦察法（铁则 4 + 铁则 12）

**问题**：Phase 4 需要对 7 个技术领域（CTP 执行、回测、风控、视频渲染、TTS、多平台发布、定时调度）做全面调研。如果串行调研，耗时会超过规划本身。

**方法**：
```
前期调查阶段 = 替补池全部 session 并行分工探查。

步骤：
1. 列出所有待调研领域（含参考项目、外部 API、内部能力、架构模式）
2. 按领域拆分为独立探查任务（每个任务 = 只读 + 单一主题 + 明确输出格式）
3. 从 session 池中选取所有空闲 session（不新建）
4. 每个 session 派发一个探查任务（包含：需读取的文件路径 + 具体问题 + 报告格式）
5. 所有探查并行运行，Commander 只接收不干预
6. 全部返回后，提炼关键决策点 → 注入规划文档

原则：
- 探查任务 = 只读（不修改任何文件）
- 每个任务有明确的"回答 X 个问题"结构
- 侦察报告标注编号（#1-#N），供后续规划 R-ID 溯源
```

**Phase 4 数据**：36 个替补 session 并行探查 7 大领域，全部返回高质量报告，每个报告含源码行号 + 架构图。

**铁则映射**：铁则 4（R-ID 贯穿全链）+ 铁则 12（分工纪律）——侦察报告编号 → 规划中 R-ID 溯源标注。

### 经验 8：三文档规划体系（铁则 4 + 铁则 9）

**问题**：Phase 3 只有一份计划文档（phase3-agent-dag-plan.md），类型定义和派发指令分散在计划中。Phase 4 将三者分离后，交叉审计发现了 20 个不一致。

**方法**：
```
标准规划产出 = 三份文档 + 一份审计报告：

1. {phase}-plan.md（主体规划）
   - 架构决策（含理由 + 备选方案）
   - 文件架构（新建/修改清单 + 行数预估）
   - R-ID 矩阵（≥20 个，5-7 波，标注并行/串行）
   - 数据流路径
   - 依赖图（Kahn 拓扑排序）
   - 测试策略 + 风险列表 + 边界条件 + 下阶段展望

2. type-contract-{phase}.md（类型契约）
   - 所有 Request/Response/DTO/trait 签名的唯一权威源
   - 标注 Rust/Python/JS 语言边界
   - 依赖矩阵（每个类型的上下游引用）
   - Remote workspace policy 注册清单

3. {phase}-dispatch-prompts.md（派发提示词）
   - 每个 R-ID 的完整 Prompt（任务→先读取→文件→步骤→验证）
   - 标注参考源（侦察报告编号 + 参考文件路径）
   - 每波标注并行/串行 + 依赖关系

4. {phase}-audit-addendum.md（审计补遗）
   - 缺口登记表（等级+根因+修复方案+所属R-ID+修复量+状态）
   - 三文档交叉一致性检查
```

**铁则映射**：铁则 4 + 铁则 9（零硬编码）——类型契约是 R-ID 之间的接口合约，三文档互相引用不重复定义。

### 经验 9：跨文档深度审计法（铁则 3）

**问题**：三份文档独立编写后如果不做交叉审计，R-ID 编号、crate 命名、类型签名会出现根本性不一致（Phase 4 审计发现 20 个缺口，4 个严重）。

**方法**：
```
跨文档审计 = 逐维度对照三份文档，找缺口。

审计维度（8 维）：
1. R-ID 一致性：计划 ↔ 派发提示词 ↔ 类型契约，数量+编号+内容是否一一对应
2. 类型契约覆盖：计划中引用的所有类型是否在 type-contract 中定义
3. 依赖图正确性：Wave 拓扑排序是否正确，有无循环依赖
4. 侦察报告溯源：每个架构决策是否有侦察报告支撑
5. 遗漏项：是否有应覆盖但三文档均未提及的内容
6. 验收标准：每个 R-ID 是否有可量化的验收标准
7. 容错设计：边界条件、降级策略是否完整
8. Phase 基线依赖：上一阶段产出是否已确认就绪

每个缺口 = {标题, 严重/中等/低, 根因, 影响, 修复方案, 所属R-ID, 修复量}
```

**铁则映射**：铁则 3（一命通关）——准备阶段的最后一步必须是跨文档审计。审计不通过 = 不开工。

### 经验 10：下一阶段前期调查纳入常规流程（铁则 3 + 铁则 5）

**问题**：Phase 2 结束后没有立即启动 Phase 3 调查，Phase 3 的 Agent 通信协议假设错误直到深度审计才发现。如果提前调查，可以减少准备阶段的审计缺口。

**方法**：
```
标准 Phase 规划的最后一步 = 下一阶段前期调查。

在当前 Phase 规划完成、审查闭合后，立即从替补池派发下一阶段的侦察任务：
1. 根据技术总纲确定下一阶段范围
2. 从参考代码地图确定待探查的外部项目
3. 从内部能力（BitFun 框架、已有 crate）确定待确认的接口
4. 复用"替补池并行侦察法"（经验 7）派发调查任务
5. 侦察结果归档，供下一阶段规划时直接引用

时机：当前 Phase 审查+终审闭合后，压缩上下文前。
```

**铁则映射**：铁则 3（准备即工作）+ 铁则 5（战至一兵一卒）——永不闲置 session 池，一个阶段结束=下一阶段调查开始。

```
│                   99%：准备阶段                                        │
│                                                                      │
│  Step 1: 前期调查（替补池全量并行侦察）                                │
│    ├─ 确定待调研领域 → 拆分为独立探查任务                               │
│    ├─ 从 session 池选取空闲 session → 派发只读探查任务                  │
│    └─ 返回侦察报告（标注 #1-#N）                                      │
│                                                                      │

│    └─ 替补池派发下阶段侦察任务（调查结果归档供下阶段引用）               │
├─────────────────────────────────────────────────────────────────────┤
│                   1%：执行阶段                                         │
│                                                                      │
│  Step 5: 拓扑派发                                                      │
│    ├─ 按 Wave 顺序逐波派发 R-ID                                        │
│    ├─ 同 Wave 内无文件冲突 → 全并行                                    │
│    ├─ 同 Wave 内有文件冲突 → 拆分亚波次                                │
│    └─ Gate Loop：每波完成后审查 → PASS/FAIL → 修正或前进               │
│                                                                      │
│  Step 6: 审查阶段（Wave N-3 ~ N-1）                                    │
│    ├─ 交叉审查 + Schema 批量校验 + 编译验证                             │
│    ├─ Bug 修复循环（根因→修复→回归→commit）                            │

│    ├─ Rust/Python/JS 代码审查                                          │
│    ├─ 文档同步                                                         │
│    └─ R-ID 逐项闭合验收                                                │
├─────────────────────────────────────────────────────────────────────┤
│                   ↳ 下一阶段前期调查（与 Step 4 衔接）                   │
└─────────────────────────────────────────────────────────────────────┘
```

**关键比例**：
- 准备阶段（Step 1-4）：99% 工作量
- 执行阶段（Step 5-7）：1% 工作量
- 审查 + 终审：嵌入执行阶段末尾（占执行阶段的 ~30% R-ID）
- 下一阶段调查：与当前阶段终审并行启动

## 十三、Phase 5 规划经验注入（v14.0.0）

### 经验 11：多方论证——侦察报告冲突时必派独立第三方裁决

当不同侦察报告对同一问题给出冲突结论时（如 B1→Zola vs B5→Hugo），不能凭偏好选一个。必须派独立 session 做第三方论证，给出明确裁决 + 切换条件。

**触发条件**：任意两个侦察报告对同一决策维度给出不同推荐方案。
**标准流程**：①识别冲突 → ②派独立 session（不读甲/乙报告，独立搜索）→ ③设定裁决维度（如维护活跃度/i18n/Rust集成/AI友好度/构建速度）→ ④产出明确裁决（不允许"视情况而定"）。
**裁决格式**：推荐 X + 否决 Y + N 个切换条件（三项同时满足才重评）。

### 经验 12：大文档并行拆分——单 session 三文档→上下文溢出

Phase 5 三文档规划总计 ~3000 行。单 session 派发时，session 需读完 8 份参考文件才开始写——上下文在上游就耗尽了（c4c501a0、3f58fb7c 均在读取阶段 /compact 死亡）。拆为 3 个并行 session 各负责一个文档后，type-contract session（e81509ad + c4c501a0）成功产出 1612 行。

**拆分策略**：Plan / Type-contract / Dispatch-prompts 各一个 session。Plan 读侦察索引 + 模板；Type-contract 读 Phase 4 类型契约 + Plan；Dispatch-prompts 读 Phase 4 派发提示词 + Plan。

### 经验 13：嵌入模板免读大文件

即使并行拆分后，session 仍可能因读取 Phase 4 模板文件（~1700 行 dispatch-prompts）而上下文溢出。最有效的解法：**把模板结构直接嵌入派发消息**（如 "格式模板（直接嵌入）：[Markdown 代码块]"），session 收到消息后可直接写，无需 Read 任何外部文件。

**适用场景**：session 收到 ≤ 当前轮次消息窗口（~180 行），无需额外 Read 即可完成写入。
**反模式**：派发消息说"先读取 X Y Z 三个文件了解格式"——这会导致 session 上下文在上游耗尽。

### 经验 14：跨文档审计是最后一道闸

三文档完成后必须交叉审计，即使各文档由不同 session 产出。Phase 5 审计发现 8 个缺口（2 严重：SSG 名称 Zola→Hugo、Crate 架构 3→1），不审计直接执行会导致 Wave 0 的 R5.0 就写错文件名。

### 经验 15：指挥官兜底——连续失败时重建规划链

当规划 session 连续失败（Phase 5 中共 5 个 session 尝试，2 个上下文死、1 个卡死、仅 1 个成功产出 type-contract），指挥官应：
1. 识别模式（连续失败 ≥ 2 次）
2. 切换策略（不再派发同类 session）
3. 直接产出（指挥官创建规划文档——属于树 1→3 方向决策，非执行）

**更新铁则**：session 连续失败 2 次→停→指挥官直接产出。

## 十四、Phase 8-10 工程实战经验注入（v15.0.0）

> 以下经验来自 Phase 8（Rebase 收尾）、Phase 9（安全审查）、Phase 10（GitHub 上架 + 质量门）的全流程。28 个 R-ID，14 个并行 Session，零回归。

### 经验 16：SessionMessage 异步派发模式（铁则 4 + 铁则 12）

**问题**：Task/AgentWait 模式下，Commander 需主动轮询等待 subagent 返回——轮询间隔浪费、超时不可控、结果混杂上下文。

**方法**：
```
SessionMessage 异步派发 = SessionControl(create) → SessionMessage(session_id, message) → 自动异步返回。

流程：
1. SessionControl(action:"create") 创建 N 个独立 session
2. 同一轮次内对所有 session 并行派发 SessionMessage
3. 不做轮询——每个 session 完成后自动回复到 Commander
4. 全部返回后，Commander 集中审查 + Gate 决策

对比 Task 模式：
- Task/AgentWait：Commander 需显式轮询，超时不可控，结果一次性涌入上下文
- SessionMessage：真正的异步推送，每个 session 独立返回，Commander 逐个处理
```

**铁则映射**：铁则 4（R-ID 贯穿全链）+ 铁则 12（分工纪律）——每个 session = 一个 R-ID 的独立执行单元。

### 经验 17：Goal continuation 静默检测（铁则 3 + 铁则 5）

**问题**：Goal 模式在 Commander turn 结束后立即触发 continuation——但此时 Commander 派发的 SessionMessage 异步回复还在路上。原检查 `queues.has_items()` 只查本地队列，不查活跃 session。

**修复**：
```rust
// 改前：仅检查本地队列
if self.queues.has_items(&session_id) { skip; }

// 改后：双条件——队列空 + 同工作区无活跃会话
let has_queued = self.queues.has_items(&session_id);
let has_active_sessions = !self.active_turns.is_empty();
if has_queued || has_active_sessions { skip; }
```

**铁则映射**：铁则 3（一命通关）——Goal 准备不充分就触发 = 违规。铁则 5（战至一兵一卒）——异步回复未归队前不推进。

### 经验 18：Rebase 工作流（铁则 3 + 铁则 11）

**问题**：main 基线前进后，不做 rebase 直接 force push 会导致 CI 失败。

**方法**：
```
标准 Rebase 流程：
1. git add -A + git commit（锁定当前所有工作）
2. git fetch origin main + git checkout origin/main 看最新 commit
3. git rebase origin/main（零冲突最优，有冲突则逐个解决）
4. 全量回归：cargo check + test + clippy + fmt
5. 如有差异：cargo fmt --all + git commit
6. 更新进度文件标注新基线
```

**铁则映射**：铁则 3（一命通关）——rebase 后不验证 = 没 rebase。铁则 11（改动前先 Read）——Read git log 确认基线变更范围。

### 经验 19：安全审查多维度并行模式（铁则 4 + 铁则 12）

**问题**：安全审查 8 个维度（Secrets、依赖供应链、路径遍历、输入验证、权限认证、日志安全、unsafe 代码、加密实践）如果串行执行耗时长。

**方法**：
```
Phase 9 安全审查模式：
Wave 1（并行审查）：8 维度全并行，全部只读，无文件冲突
  ├─ R9.1 Secrets 考古（grep API key/secret/token 模式）
  ├─ R9.2 依赖供应链（cargo audit + 非 crates.io 依赖审查）
  ├─ R9.3 路径遍历 + SSRF（审计文件路径拼接、URL 构造）
  ├─ R9.4 输入验证（tick 数据边界、反序列化安全）
  ├─ R9.5 权限认证（命令执行、令牌存储）
  ├─ R9.6 日志安全（敏感数据泄露）
  ├─ R9.7 unsafe 代码（grep unsafe）
  └─ R9.8 加密实践（哈希/随机数/加密库）
  └─ R9.11 SECURITY.md（可与审查并行）

Wave 2（串行修复）：R9.9 P0 修复 → R9.10 P1 修复
  - P0 优先：路径遍历 canonicalize、SSRF redirect none、微信 secret URL 泄露
  - P1 其次：NaN/Inf 传播、负价格校验、除零保护、命令注入 sanitize

Wave 3（归档）：R9.12 安全报告（覆盖 12 R-ID 闭环）
```

**铁则映射**：铁则 4（R-ID 贯穿全链）+ 铁则 12（分工纪律）——审查全并行、修复串行（按风险等级），Gate 逐波闭合。

### 经验 20：质量门清单（铁则 3）

**问题**：代码写完后直接 force push = 把验证负担全部甩给 CI。应该在本地跑完整质量门。

**方法**：
```
最终交付前的必检清单（Phase 10 Wave 3）：

| 门 | 命令 | 标准 |
|----|------|------|
| 编译 | cargo check --workspace | 0e 0w |
| 测试 | cargo test -p <all taiji crates> | 全部通过 |
| Clippy | cargo clippy --workspace | 0 warning (taiji) |
| 格式 | cargo fmt --check --all | passed |
| 卫生 | pnpm run check:repo-hygiene | passed |
| CI 配置 | pnpm run check:github-config | passed |
| 安全报告 | docs/plans/phase9-security-report.md | 已归档 |

顺序：check → test → clippy → fmt → hygiene → CI-config → 安全报告
任何一门失败 → 修 → 回到第一门重跑
```

**铁则映射**：铁则 3（一命通关）——缺任何一门质量门 = 不交付。

### 经验 21：已知 bug 不得遗留（铁则 3 + 铁则 6）

**问题**：Phase 8 进度文件登记了 4 个 🔴 上游 bug（DAG 重复边、panic 隔离、CTP placeholder、kline_renderer NaN），标注"上游已有"准备跳过。但"做到极致"不容忍已知 bug。

**方法**：
```
Bug 修复原则：
- "上游已有"不是理由——既然是另起炉灶，就要修到底
- 每个 bug 派一个独立 session：读代码→定位根因→最小修复→加测试→验证
- 修复后回写缺口登记表

Phase 10 修复结果：
- DAG: add_edge 去重 + debug_assert + 单元测试
- Panic: catch_unwind 隔离每个 node.process
- CTP: connect/disconnect 返回明确错误替代静默 Ok
- kline_renderer: NaN/Inf 前向填充 + 指标 skip + 6 新测试
```

**铁则映射**：铁则 3（一命通关）+ 铁则 6（防呆设计）——已知会崩的路径必须修，不能"等用户遇到再说"。

### 经验 22：GitHub 上架文档清单（铁则 9）

**问题**：仅有代码没有配套文档 = 无法上架 GitHub。

**方法**：
```

├── CONTRIBUTING.md / CONTRIBUTING_CN.md（加入 taiji 贡献指南）

├── SECURITY.md / SECURITY_CN.md（taiji 安全策略）
├── 每个 crate 的 README.md（含闭源 crate 的 placeholder 提示）
├── .github/workflows/ci.yml（taiji 专属 CI job：check/test/audit/clippy）
└── docs/plans/phase9-security-report.md（安全审查报告）
```

**铁则映射**：铁则 9（零硬编码）——文档是配置的一部分，不上架 = 不完整。

## 十五、Phase 1-7 历史经验注入（v16.0.0）

> 以下经验来自 Phase 1-7 共 24 个 session（批次 A 01-12 + 批次 B 13-24）的全量方法论分析，加上早期指挥官 session（10 个）的战略决策模式提取。每条经验绑定对应铁则。**不可重复 v15.0.0 已有的经验 16-22。**

### 经验 23：五层派发模型（铁则 4 + 铁则 12）

按任务确定性选择派发格式。确定性越高，Commander 给出越多细节；不确定性越高，Agent 自主空间越大。

| 层级 | 格式 | Agent 角色 | 适用场景 |
|------|------|-----------|---------|
| L1 | 完整代码嵌入 | 写入器 + 编译验证器 | 骨架创建、类型定义、简单 trait |
| L2 | 关键签名 + 验收标准 | 签名锁定下的实现者 | 模块实现、trait 实现 |
| L3 | 结构化派发（优先级+依赖+参考三行元数据） | 自主执行者 | Phase 6 大规模执行、跨语言任务 |
| L4 | 侦察派发（关键词+问题列表+产出格式） | 调研者 | 技术选型、外部代码分析 |
| L5 | Meta 文档生成（严格参照已有格式） | 文档生成器 | 跨 Phase 派发文档、类型契约 |

**核心**：Commander 的任务确定性感知决定派发层级——能直接写的绝不派发调研，需调研的绝不假装知道答案。

### 经验 24：三段式 Session 结构 + Turn 0 锚点（铁则 4）

每个 session 遵循严格三段式：**Turn 0 锚点层**（创建可被后续引用的类型/trait，1-3 个文件，编译验证通过）→ **Turn 2 侦察层**（外部代码分析，L4 格式）→ **Turn 4+ 执行层**（R-ID 实现，确定性递减）。

Turn 0 的锚点作用：确保每个 session 第一步产出是"可被后续引用"的类型/trait——避免 session 空转等待其他 session 产出依赖。

### 经验 25：共享核心类型预暴露只读 API（铁则 9）

StateStore / Pipeline / Signal 等共享核心类型在设计阶段就预暴露公共只读访问 API（如 `state_store_arc()`、`pipeline_status()`）。每次后来者需要"反向开孔"暴露私有字段，累计浪费的编译迭代远超提前暴露的成本。教训：共享类型的设计必须预判下游消费需求。

### 经验 26：字段变更 workspace 级全量验证（铁则 3 + 铁则 11）

共享 struct 增字段后，必须执行 `grep 'StructName {'` workspace 级全量构造点搜索——单 session 内不可能覆盖所有构造点（闭源 crate 不在当前编译单元首轮检查范围内）。建立"字段变更 checklist"：增字段→grep 全量构造点→一次性修复→workspace 级 cargo check + test。

### 经验 27：R-ID 隐式拓扑编排（铁则 4 + 铁则 9）

Commander 用 R-ID 序列管理跨 session 依赖拓扑，Agent 不需知道其他 session 的存在。Phase 4 类型层纯并行（无互相依赖），Phase 5 开始跨 crate 引用，Phase 6 P0 任务可提前在 Phase 5 执行期启动——缩短关键路径。原则：同层无依赖→全并行；跨层有依赖→串行但提前启动 P0。

### 经验 28：参考链标准化——可追溯决策（铁则 10 + 铁则 11）

从单级引用（type-contract → 代码）演进为压缩链：**侦察编号（如 B3）+ Plan 决策编号（如 Plan 决策 11）→ 类型签名确认（先读确认）→ 实现**。每个实现决策可追溯至原始侦察报告，防止"为什么这么做"失传。侦察编号是决策的不可变溯源 ID。

### 经验 29：Recon 不空转——侦察→执行闭环（铁则 3）

每个侦察报告必须在后续至少一个 Execute 任务中被引用（侦察编号出现在实现 R-ID 的"参考"字段中）。Commander 只在有明确执行计划时才发起 Recon。侦察报告格式强制四段式（关键发现 + 推荐方案 + 备选方案 + 风险），≤ 1 页。

### 经验 30：Python→Rust 移植五步法（铁则 10）

① 读取 Python 源文件精确行范围（不做猜测）→ ② 逐行对比 Python vs Rust 逻辑，标注差异点，确认"有意且自洽"→ ③ 解决依赖（检查 workspace Cargo.toml，使用 `workspace = true`）→ ④ 写实现 + 至少 3 个测试（空输入/正常数据/边界）→ ⑤ 编译验证 + 修复警告 + 测试通过。此法在 3 个 session 中成功移植了 `find_pivots_raw`、`find_triple_push`、`calc_magnet`。

### 经验 31：通用实现五步法——PRE-FLIGHT 到 CONVENTION-DETECTION（铁则 3）

**PRE-FLIGHT**（读参考文件，确认类型/依赖存在，评估并发冲突）→ **BLOCKER-FIRST**（补齐缺失依赖，修复旁路编译错误，再写主逻辑）→ **SKELETON→FLESH**（先创建可编译骨架含 TODO，逐步用实现替换）→ **VERIFY-EVERY-STEP**（每次变更后立即 cargo check/build/test，不积累错误）→ **CONVENTION-DETECTION**（搜索已有代码模式，遵循既有约定，不盲从字面指令）。

### 经验 32：三级问题分类法——编译失败先分类（铁则 5）

编译失败时不盲目修复。先分类：① 自己的代码（直接修复）→ ② 直接依赖的预存问题（选择性修复，记录原因）→ ③ 无关 crate 的问题（标记跳过，不蔓延范围）。避免在无关问题上消耗时间，防止范围蔓延和"修 A 坏 B"。

### 经验 33：L1 代码不是免检的（铁则 11）

即使 Commander 提供的完整代码（L1 格式）也需要与现有类型契约做比对——复用已有类型、消除 lint 警告、适配缺失 derive。L1 不意味着"复制粘贴即可"，仍然需要执行完整的编译验证流程。

### 经验 34：cargo check ≠ cargo test 高频失败模式（铁则 3）

`cargo check` 不编译测试代码，check 通过 ≠ test 能编译。批C 中发生 5+ 次：check 零错误但 test 编译失败（测试代码引用未导出类型、测试 mock 缺少 trait 方法等）。**固定验证流程**：先 `cargo check` → 再 `cargo test`，不可跳过第二步。

### 经验 35：根因收敛原则——不逐条修补（铁则 3）

多 bug 常共享同一根因。不逐条修补表层症状——定位共享根因，一次修复解决全部。Phase 9 安全审查从 100+ 条告警收敛到 4 条根因修复。验证形式化陷阱：测试通过 ≠ 行为正确（如测试只验证 happy path，边界条件未覆盖），需额外执行路径追踪审计。

### 经验 36：三版架构分层 + Trait 边界 = IP 隔离墙（铁则 7 + 铁则 9 + 铁则 10）

**三版架构**：太极 dev 版（热更新开发）→ SRC 稳定版（build 固化，非原版而是中间产物版）→ 官方原版（参考对照）。三个独立工作区，各自明确用途，互不污染。

**Trait 边界**："Engine defines WHAT (traits), Strategy defines HOW (impls)"。开源部分：所有 trait 定义、Pipeline 引擎、数据管道、通用算法。闭源部分：具体策略实现、量化参数、太极公式。移除闭源 crate 时注释保留路径——"加法不做减法"。

### 经验 37：侦察驱动决策 + 数据规模决定方案（铁则 1 + 铁则 9 + 铁则 10）

**侦察驱动**：所有外部依赖假设必须在实现前通过代码侦察验证。BitFun CronService 文档声称存在但代码中不存在 → 立即侦察→评估影响→切换替代方案（TaskDag）。教训：文档 ≠ 代码，侦察先行，不凭假设决策。

**数据规模第一**：技术选型的第一输入是数据规模，不是功能列表。四总纲 < 2,000 行 → Grep + LLM 重排序足够，不需要 +50MB 向量数据库。不过早引入重量级依赖。
