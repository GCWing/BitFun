# 洞察汇报（Insight Report）

## 风格标识
`insight-report`

## 适用场景
学术汇报、行业研究、咨询分析、内部洞察/复盘汇报、评测与尽调报告、技术方案深拆、竞品/方案对比分析等需要详尽论证与高信息密度的工作/研究场景演示。

## 风格定位
这是「给人看的分析报告」，不是「给人讲的演讲幻灯」，更不是「raw 数据的堆砌上墙」。读者是会前后逐行细读的领导、评审和同行，每一页都必须经得起脱离演讲者的独立阅读。

**核心理念：把 raw 数据转化为有效信息，并用合理的视觉布局提升阅读者的阅读效率。** 优先级排序：**有效信息密度 > 逻辑显性 > 版面美观**。其中「有效信息密度」不等于「文字字数」——一张精准的对比图传递的有效信息，远胜过三段描述同一组数字的文字。留白不是美德，但**用低信息量的文字填满页面同样不是美德**；真正的目标是让读者用最短的视觉路径获取最强的判断。

## 三条核心原则（最高优先级，贯穿全 deck）

### 原则 1：数据 → 信息 → 判断，三级递进
- **raw 数据**（数字、事实）本身没有阅读价值；必须先转成**信息**（数字之间的对比、构成、趋势、差异），再收口到**判断**（so what）。
- 每个页面都要完成这条链路。禁止只停留在 raw 数据陈列（一张纯数字表格 + 「数据来源」脚注 = 不合格）；禁止只有判断没有证据（空洞的结论句）。
- **对比类数据是可视化的第一优先对象**：当页面核心是「A vs B」或「多对象多维度对照」时，对比图（双柱、标尺、雷达、热力矩阵）几乎总是比纯文字表格更有效。

### 原则 2：形式服务于信息，而非信息迁就形式
- 版式、图表、色系、字号的选择，唯一判据是「哪种形式能让读者最快、最准确地 get 到本页判断」。
- **不允许用固定模板套所有页面**：即使内容是六个维度的对比，也不应该六页都用「左右双栏卡片」。根据每页的信息意图选版式（见「版式选择」）。
- 文字、表格、图表、流程图、结构化图示都是合法表达；**混合使用且节奏多变**才是成熟报告的样子。

### 原则 3：有效信息密度，而非字数密度
- 本风格鼓励高密度，但「密度」的度量是**单位面积内的有效信息量**，不是汉字个数。
- 一张承载 5 组对比双柱 + 标注的图表区，其有效信息密度远高于等面积下 200 字的纯描述段落——前者一眼得结论，后者要逐行读才能捞出数字。
- 因此：**该图则图，该表则表，该文则文**。纯文字段落适合「机制解释、因果论证、定性判断」；数字对比、构成、趋势、多维画像适合图表；步骤、架构、因果适合结构化图示。

## 信息可视化（硬约束，不再是软建议）

> 参考 `references/data-information-visualization.md` 的完整词汇库。本节是该参考在 insight-report 风格下的**强制执行规则**。

### 必须可视化的页面类型
凡页面主要论证手段属于以下类型，**必须至少包含 1 个非纯文字的视觉表达元素**（对比图、构成图、趋势图、流程图、架构图、热力矩阵、评分标尺等）。纯文字段落 + 纯文字表格不得作为该类页面的唯一载体：

| 页面信息意图 | 必须包含的视觉元素（至少其一） | 禁止的退化形态 |
|---|---|---|
| 数值对比（A vs B、排名） | 对比双柱、水平条形、评分标尺、点图 | 仅用文字描述「A 是 1410，B 是 50」 |
| 构成占比 | 堆叠条、100% 构成条、环形（仅可编辑路径外） | 仅用文字「国企占 73.16%」 |
| 趋势 / 阶段演进 | 折线、柱状、斜率图、时间线/阶段带 | 仅用文字罗列时间点 |
| 多维画像 / 综合对比 | 雷达图、热力矩阵、小倍图、对比矩阵 | 仅用文字逐维度描述 |
| 流程 / 系统 / 因果 | **分层架构图**（系统组成）、**线性流程图带箭头**（处理步骤）、**泳道图**（多角色协作）、**因果链/问题树**（根因分析） | 仅用文字描述步骤或系统组成 |
| 层级 / 决策 | 树状图、金字塔、决策矩阵、2×2 | 仅用文字 bullet 列表 |

### 数据可视化的质量要求
- **每张图必须直接服务于本页标题结论**，而非只是让页面「看起来有数据」。标题说「X 方案在 Y 维度碾压」，图就必须让这个碾压一眼可见。
- 图表必须有：数值标签、单位、时间/口径、来源。没有可靠数字时用定性矩阵或结构化图示，**不编造数据、不伪造精度**。
- **对比主体稳定着色**：若 deck 存在贯穿始终的对比主体（如 A 校 vs B 校、方案 X vs 方案 Y），必须在第一页为其分配固定主题色，后续所有页面（栏头、数据条、图表系列、图例）复用同一色系，形成视觉锚点。读者翻到任何一页都能 0 成本分辨「谁是 A、谁是 B」。
- 强调色引导结论：关键差异、胜负、风险点用强调色（红）标出；但全页不得有超过 3 处独立的红色强调，否则失去焦点。
- **可编辑 PPTX 兼容**：条形/柱形/进度/堆叠条用纯色 `div`；折线用细 `div` + `transform: rotate()`；热力矩阵用离散 3–5 档纯色；不用 `conic-gradient`、复杂 SVG path、连续渐变（除非明确只做 1920 演讲版）。

### 图表类型多样性（硬约束，防止单一图表类型垄断全 deck）
- **同一 deck 内，图表类型必须多样化**：整套 deck 至少出现 **3 种以上不同图表类型**（如水平条形、雷达/多维矩阵、堆叠构成条、时间线/阶段带、决策矩阵、流程节点、热力矩阵、点图等）。只用 1–2 种图表类型（哪怕每页都画了图）= 视觉单调，不合格。
- **连续两页不得使用相同图表类型**：若上一页用了水平条形对比，下一页必须换一种（如换雷达、堆叠条、时间线）。这跟「版式多样性」是双重约束——版式不同但图表相同，仍算单调。
- **按信息意图匹配图表类型，而非按惯性**：这是最容易犯的错——模型学会一种图（通常是水平条形对比柱）后，对所有对比类页面都用它。正确做法是逐页判断该页数据的结构，选择最匹配的图表：

| 该页数据结构 | 最匹配的图表类型 | 不要总是退化为 |
|---|---|---|
| 2 对象 × 少量数值（2–4 个） | 水平对比条形 / 双柱 | ✅ 合适 |
| 2 对象 × 多维度（5+ 维）综合画像 | **雷达图**（多维轮廓一眼对比） | ❌ 不要退化成多组条形或大表格 |
| 2 对象 × 多维度 × 精确数值查阅 | **热力矩阵**（格内带数值 + 颜色档） | ❌ 不要退化成纯文字表格 |
| 单对象或双对象的构成占比 | **堆叠构成条 / 100% 条** | ❌ 不要用文字「占 73%」 |
| 有时间演进 / 阶段递进 | **时间线 / 阶段带**（节点 + 标注） | ❌ 不要用纯文字段落罗列 |
| 决策 / 分类建议 | **决策矩阵 2×2 / 决策树** | ❌ 不要用小 chip 拼贴或长表格 |
| 流程 / 系统 / 因果 | **流程节点图 / 架构图** | ❌ 不要用文字编号列表 |
| 离散对象的位置关系（如高/低 × 大/小） | **点图 / 2×2 散点定位** | ❌ 不要用表格 |

> **最常见失败模式**：模型对每一个「A vs B」页面都画一组蓝红水平对比柱，8 页里有 4 页长一样的图。这会让读者视觉疲劳，且错失了用雷达/构成/时间线等更精准表达的机会。**每选一种图表前，先问：这个数据结构真的是"两个数比大小"吗？如果是多维度、构成、时间、决策，就该换图表类型。**

## 版式多样性（硬约束）

### 版式选择规则
本风格定义 6 种主版式（见下方「推荐版式」）。生成 deck 时必须遵守：

1. **按内容意图选版式**，不按惯性选。每页根据其主要信息意图（对比/构成/趋势/流程/深拆/综述）匹配最合适的版式。
2. **连续两页不得使用相同主版式**。若两页内容确实同质，应合并为一页或拆换其中一页的版式。
3. **整套 deck 至少覆盖 3 种以上不同主版式**。8 页 deck 只用 2 种版式 = 不合格。
4. 每选定一种版式，必须达到该版式的量化最低规格（行数/列数/节点数），见下方。

### 对比类 deck 的结构建议
当 deck 主体是「多对象多维度对比」（如 A vs B、三方案选型）时，推荐结构：

1. **封面**：摘要 + 核心对比指标（用对比双柱/标尺，禁用纯文本 metric 列表）
2. **总览页**（强烈推荐）：一张全维度对比矩阵或雷达图，概括所有维度的胜负，给读者全局地图
3. **逐维深拆页**（每个维度一页）：每页聚焦一个维度，用该维度最适合的版式
4. **结论页**：综合判断 + 决策建议（用决策矩阵/决策树，禁用小 chip 拼贴）

## 画布填充硬指标（写时遵守）

- **垂直填充率 ≥ 90%**：正文内容必须从标题分隔线下方一直排到页底脚注线上方。若内容结束位置距脚注线还有超过 40pt 的空白，该页不合格——继续增加模块、行、列或展开分析，直到填满。
- **有效内容量目标**：每页应承载足够的论证支撑这个判断。「足够」的度量是**信息量**而非字数——一组 5 维对比雷达图 + 3 条标注 + 200 字解读，和 600 字纯文字段落，前者信息量通常更高。
- **纯文字内容页**：正文 450–800 个汉字（或等量英文）为目标区间；低于 300 字的纯文字内容页不合格。
- **图文混合页**：文字量可相应减少（图表已承载大量信息），但图旁的解读文字不得少于 150 字——图不能「自己说话」，必须有文字点出 so what。
- **不许留白，但也不许凑数**：素材不足以填满一页时，按以下顺序补密度（注意是补「有效信息」不是补「字」）：
  1. 增加可视化（把已有数字转成对比图/构成图）
  2. 增加分析维度（风险、反例、边界条件、竞品对照）
  3. 给每条结论补具体证据（数字、日期、文件、原话）
  4. 增加「So what / 本页结论」横条
  5. 与相邻主题合并成一页
- **禁止用无效数据凑行**：对比表里某行双方都「数据未公开」或数值完全相同 → 该行无信息量，必须删除或替换为有差异的指标。
- 每页底部、脚注线之上，放一条**「本页结论」横条**（浅灰底或左侧主色粗边线，1–2 个完整句子），既补密度又强制每页有判断。封面与总览页除外。

## 信息密度与语言

- 用**完整句子**写分析，禁止电报式名词短语。每条要点都是「主语 + 判断 + 限定/证据」、可独立阅读的句子。
- **每个卡片/单元格/小节都必须是多行实体**：卡片至少 4–7 行（一个加粗小标题 + 3 组以上「标签：完整句子」字段）；只有 1–2 行的卡片不合格，要么补字段要么合并。
- 鼓励 2–3 级嵌套列表：第一级是阶段或分论点，第二级是固定分析维度，第三级是具体证据或例子。
- 行内强调系统承担「快速扫读」功能：
  - 关键概念、机制、判断词**加粗**（`font-weight: 700`）；
  - 失败、风险、转折点、用户介入等警示信息用强调色（可配斜体）；
  - 代码标识符、文件名、函数名用等宽字体。
  - 读者只扫加粗与红字就能复述页面骨架；逐行细读则获得完整论证。两层都必须成立。
- 引用原话、指令、报错时直接嵌入正文并加引号或等宽字体，不要省略成「用户提出修改意见」这类空话。
- 数字必须带口径与时间（「2024 年某指标 N 人，全国第一」式），孤零零的数字不算证据。

## 逻辑骨架（每页的分析结构必须显性、可复述）

- **固定分析框架复用**：从内容中提炼一组分析维度，做成带冒号的加粗标签前缀，在页内每个小节按同一顺序重复出现。例如「**定位：**… / **规模：**… / **优势：**… / **短板：**…」。同一页内框架必须一致，让读者按同一把尺子对比各小节。
- **过程叙事页**：按时间线交替展开「动作 → 反应」（如 Prompt1 → 行为1 → Prompt2 → 行为2 …）；节点之间插入强调色批注句标记转折点；页首放元信息块（工具/对象/任务/轮次等键值对），页侧放结果指标面板。
- **评测/对比矩阵页**：满版大表格或热力矩阵是页面主体而不是配图。单元格内放多行内容：评分加粗 + 编号小结论列表；失败、异常单元格整体用强调色标出；首列加粗，表头深色底白字。**但当对比维度 ≤ 4 且核心是「谁赢」时，雷达图或评分标尺比表格更直观**，应优先考虑。
- 标题写法：「主题 + 判断」（如「维度X：主体A vs 主体B，各自的核心优势」）或「主题 – 切面」。正文与图表必须逐条支撑标题。
- 全篇组织：总-分结构。先一页结论总览（推荐用满版矩阵或雷达图承载），后续每页深拆一个案例/维度/论点，最后一页给出可执行的启示或建议。

## 推荐版式与量化最低规格（960×540pt，每种版式必须达标）

### 1. 满版评测矩阵 / 热力矩阵
适用：多对象 × 多维度的精确对照、排名、评分汇总。
- 表格 **5–11 列 × 7–12 行**，或热力矩阵（对象 × 维度，每格 3–5 档颜色 + 关键数值）。
- 信息量大的单元格放「加粗短判断 + 编号小结论（1–3 条）」；失败格标红。
- 行数不够就增加对比维度行，或在表格下方补一栏「关键发现」编号列表直至填满。
- **若维度 ≤ 4 且对象 ≤ 5，优先考虑雷达图总览 + 表格明细的组合**，比纯表格更一眼。

### 2. 卡片网格深拆
适用：多个并列对象/案例的同等粒度拆解。
- 2×3 或 3×3 网格撑满整个正文区高度，**每张卡片 4–7 行**：加粗小标题 + 3 组以上「**标签：**完整句子」字段。
- 六张两行卡片只占半页属于典型不合格输出。
- **对比主体卡片须用稳定主题色区分**（栏头底色或左侧色条）。

### 3. 对比图 + 解读（insight-report 核心版式）
适用：页面核心是数值对比、构成对比、趋势对比。
- 上方或主区放对比图（双柱、堆叠条、评分标尺、雷达），占正文区 50–65%；
- 下方或侧栏放 3–5 条解读：每条「加粗判断 + 证据数字」，点出图表里的关键差异和 so what。
- **对比主体的色系必须全 deck 一致**。
- 禁止只有图没有解读（图不会自己说话），也禁止只有文字描述没有图（对比数据必须可视化）。

### 4. 三级嵌套深拆页
适用：因果论证、机制解释、定性分析（无强数值对比）。
- 3–4 个阶段/分论点小节，每节内按同一组加粗标签维度展开完整句子分析，每节 4–8 行，整页几乎全文字，从分隔线排到脚注线。

### 5. 过程叙事 + 侧栏
适用：时间线事件、对话/交互复盘、阶段演进。
- 左侧细边框容器装时间线叙事（顶部元信息键值块 → Prompt/行为交替段落 → 红色批注句），节点 4–6 个；
- 右侧窄栏「效果/结论」面板 5 行以上（标签 + 数值/判断）。
- 也可演化为**时间线/阶段带 + 下方解读**的横向布局。

### 6. 流程/架构图深拆
适用：系统组成、责任分工、因果链、决策路径。
- CSS 绘制的多级流程图或架构图占约 55–60%，节点 6 个以上且带标注；
- 旁边一栏逐节点完整句子解释，与节点编号一一对应。
- 结论页的「决策建议」适合用决策树/决策矩阵形态，而非小 chip 拼贴。

### 7. 左右对照深拆
适用：两个对象的深度并列对照（非数值对比，而是定性基因/定位对照）。
- 左右两栏各装一个多字段实体卡（各 6–10 行），下方横排 1–2 条红色「关键风险/核心疑问」批注条，再加「本页结论」横条收底。
- 两栏只占上半页、下半页留白属于不合格输出。
- **左右栏须用对比主体稳定色系区分**（如左蓝右红/左蓝右紫）。

## 封面页特殊规范

封面是「一眼抓住结论」的页，不是「大标题 + 半页空白」，也不是「信息很多但记不住」。

- 左侧：120–250 字摘要段（分析背景 + 框架 + 核心结论预览）。
- 右侧：**核心对比指标区**，禁用纯文本「标签 + 值」两列列表。至少有 1–2 个指标用**对比双柱 / 评分标尺 / 迷你对比条**呈现，让差距量级一眼可见。
- **指标选数原则**：只选「有对比张力」的指标（双方差距大、或互有胜负）。双方数值相同（如都是「1 个重点实验室」）的指标不得放入封面；重复或近义的标签必须合并。
- 底部 conclusion-bar：一句话定位（「A 是 X，B 是 Y」式）。

## 视觉系统
- 纯白背景（#ffffff），零装饰：没有大色块、渐变、图标堆砌和情绪化视觉；**信息本身就是视觉**。
- 顶部标题区一行大字（允许两行），与正文区用细分隔线分开；左上角放章节 kicker（如 03 | 案例深拆），右上角页码。标题区总高度不超过画布的 18%，把空间留给正文。
- 正文承载区占画布 80% 以上。常用分区：单栏全宽（满版表格/嵌套列表/大图）、主区 + 侧栏（约 70% + 25%）、2–3 栏对称拆解。
- 内容容器只用细边框（1–1.5px 实线）和浅灰底面板，不用投影、大圆角、玻璃拟态。
- 页面底部保留来源/口径/假设脚注行（font-size: 9–10px）。

## 配色（CSS 可用）
- 背景：#ffffff（纯白）
- 文字：#1f2937（深灰黑）
- 主色：#1e3a8a（深海军蓝：标题、表头、结构线、编号、标签前缀）
- 强调：#dc2626（红色：失败/风险/转折批注、关键差异、胜负标记，克制使用但该用就用）
- 辅助：#64748b（中灰蓝：副标题、说明文字、脚注）
- 面板：#f1f5f9（浅灰蓝：表头底色、元信息块、侧栏面板、斑马纹）
- **对比主体色（当 deck 存在贯穿对比时必须定义）**：在封面为各对比主体分配稳定色，后续所有页面复用。一旦定义，全 deck 不得换色。

### 色彩纪律（防止配色混乱，硬约束）
全 deck 的**彩色（非中性灰）总数不得超过 3 个**：主色 + 1 个对比主体色 + 强调红。这是「高级感」的来源——色彩越克制越显专业。具体规则：

1. **对比主体 B 色必须与主色蓝有足够色相距离**。主色蓝是 `#1e3a8a`（冷色、深色），对比主体 B 应选**暖色系或对比色**，**不要选紫色系**（`#7c3aed` 等）——紫色与深蓝色相太近，放在一起像「深蓝 vs 蓝紫」，区分度不够反而显脏。
   - ✅ 推荐 B 色：`#be123c`（深红/酒红，与蓝形成冷暖对比，最经典）、`#b45309`（琥珀/深棕，沉稳）、`#0f766e`（墨绿，与蓝形成补色关系）。
   - ❌ 禁用 B 色：`#7c3aed`（紫，与蓝太近）、`#9333ea`（亮紫，花哨）、任何与 `#1e3a8a` 色相差 < 120° 的冷色。
2. **热力矩阵 / 构成条等需要深浅档位的图表，深浅档必须用「主体色的同色系明度变化」或「中性灰系」，不得引入新的彩色**。
   - ✅ 正确：主体 A 档 = `#dbeafe / #93c5fd / #1e3a8a`（蓝的同色系三档）；主体 B 档 = `#fecaca / #f87171 / #be123c`（红的同色系三档，若 B 是红）。
   - ❌ 错误：主体 A 用蓝三档 + 主体 B 用紫三档 = 6 个彩色同时出现，画面立刻混乱。
   - 若对比主体 B 是红，则热力矩阵中主体 A 用蓝系深浅、主体 B 用红系深浅，总共仍是「蓝 + 红」两个彩色系，只是各有明度变化——这是可接受的。
3. **强调红 `#dc2626` 只用于风险/失败/关键差异标注**，不作为对比主体色（否则与红色系的主体 B 冲突）。若对比主体 B 本身就是红色系，则强调标注改用加粗 + 斜体，不再额外用红色。

## 排版（HTML/CSS）
- 正文 12–14px，line-height 1.45–1.6；嵌套列表逐级缩进 14–18px；段间距保持密排（不超过 1.5 倍行距），整页呈现「满而有序」。
- 表格字号 11–13px，单元格 padding 4–8px；单元格内的编号列表用 `margin: 0; padding-left: 14px;` 紧凑排布。
- 结论标题 20–24px 粗体；小节标题与侧栏面板标题 13–15px 加粗。
- 数字、评分、代码用等宽字体（如 "SF Mono", Consolas, monospace）增强报告感。
- 图表标注：数值标签 11–13px 加粗；轴标签/图例 10–11px 中灰；关键差异标注用强调色。

### 字号下限（硬约束，不可妥协）
全页**任何可见文字的 font-size 不得低于 10px**（包括图表内的数值标签、图例、时间线卡片正文、堆叠条内文字、热力矩阵单元格文字、脚注）。这条规则高于「画布填充率」「正文密度」等所有其他指标。

常见违规场景与正确做法：
- **堆叠构成条/对比条里塞不下文字** → 文字不要硬塞进条内，改成标签写在条外侧，或拆成两行。
- **时间线/雷达卡片正文太长塞不下** → 不要把字号压到 8px；精简每张卡片到 2-3 行，或减少卡片列数（4 列→3 列）。
- **热力矩阵单元格文字密集** → 单元格字号保持 11px，减少表格行数或把长描述移到表格下方的解读区。

### 拆页优于缩字（优先级，硬约束）
当内容装不下时，**必须拆成两页或减少信息模块，绝不通过缩小字号来硬塞**。判断顺序：
1. 先尝试减少次要模块（如删掉冗余解读卡、合并近义维度）。
2. 再尝试把本页拆成两页（每页聚焦一半内容，各自保持可读字号）。
3. **最后才考虑微调（且不得低于 10px 下限）**——这是下策，且只能在图表标注/脚注这类辅助文字上用，正文永远 ≥ 12px。

「内容装得下但排不满」时反过来：按「画布填充硬指标」补有效信息（优先补可视化，其次补分析维度）。

- 垂直预算心算（写前必做）：标题区 70–95pt + 脚注 20–25pt + 底部安全边距 ≥ 36pt（0.5in），正文区实际约 390–420pt。每个块按「行数 × 字号 × 行高 + padding」估算，图表按「高度 + 标注区」估算，所有块合计 ≤ 正文区高度；表格先算「行数 × 行高」再定行数。估超就减行/合栏/拆页，禁止缩字号硬塞。

## CSS 实现要点

### 文字与结构
- 标签前缀：`<strong style="color: #1e3a8a;">定位：</strong>` 后接常规正文，同一页所有小节复用同一组标签。
- 批注条（风险/疑问/转折）：`background: #fef2f2; border-left: 3px solid #dc2626; padding: 6px 10px;`，内文 `color: #dc2626;` 可配斜体。
- 「本页结论」横条：`background: #f1f5f9; border-left: 4px solid #1e3a8a; padding: 8px 12px;`，1–2 个完整句子。
- 元信息块/侧栏面板：`background: #f1f5f9; border: 1px solid #e2e8f0; padding: 10px 12px;`，内部用「加粗标签: 值」逐行排布。
- 流程框：`border: 1.5px solid #1e3a8a; border-radius: 4px; padding: 6px 10px;`，连接线用细 div 或 border 绘制，箭头用 CSS 三角形。

### 表格
- `border-collapse: collapse;`，表头 `background: #1e3a8a; color: #ffffff;`，斑马纹 `tr:nth-child(even){background: #f1f5f9;}`，单元格 `border: 1px solid #e2e8f0; padding: 4px 8px; vertical-align: top;`。
- 多行单元格：单元格内用 `<ol>`/`<ul>`（`margin: 0; padding-left: 14px; font-size: 10px;`）放编号小结论；失败格 `color: #dc2626; font-weight: 700;` 或整格 `background: #fef2f2;`。

### 纯 CSS 图表工具箱（可编辑 PPTX 兼容，直接套用）

> **使用纪律**：以下 8 种图表对应不同的数据结构，**必须按页面的信息意图选择最匹配的一种，不要对所有对比页都用「对比双柱」**。参考上方「图表类型多样性」表的匹配关系。每张图的对比主体色必须与全 deck 定义一致。

#### ② 雷达图近似（2 对象 × 5–6 维综合画像，适合「整体轮廓对比」）
纯 CSS 无法画真正的多边形雷达，但可用**同心多边形表 + 逐维双柱**实现等效的「多维一眼对比」。每个维度一行，用两条短柱表达双方在该维的相对位置（归一化到 0–100），读者扫一列就能看出谁强谁弱：
```html
<div style="display:flex; flex-direction:column; gap:6px;">
  <!-- 每个维度一组：维度名 + 双方相对位置条 -->
  <div style="display:flex; align-items:center; gap:8px;">
    <p style="width:84px; font-size:11px; font-weight:700; color:#1f2937; margin:0;">维度名</p>
    <div style="flex:1; display:flex; flex-direction:column; gap:2px;">
      <div style="display:flex; align-items:center; gap:6px;">
        <div style="width:96px; height:12px; background:#1e3a8a;"></div><!-- 主体A 80% -->
        <p style="font-size:11px; color:#1e3a8a; margin:0;">80</p>
      </div>
      <div style="display:flex; align-items:center; gap:6px;">
        <div style="width:120px; height:12px; background:#be123c;"></div><!-- 主体B 100% -->
        <p style="font-size:11px; color:#be123c; margin:0;">100</p>
      </div>
    </div>
  </div>
  <!-- 重复 5–6 个维度（从输入内容中提炼，如规模/经费/产出/口碑/覆盖度…）-->
</div>
```
> 这是「small multiples 对比条」形态，视觉上等同于把雷达图的每根轴拆成一行双柱。比纯表格更一眼，比真雷达图更可编辑。

#### ⑤ 时间线 / 阶段带（时间演进 / 阶段递进，适合「未来怎么走」）
> **布局纪律**：不要用「每列内 dot + `flex:1` 竖线 + body」的拉伸结构——当容器高度不确定时竖线会塌陷。改用**三段式固定结构**：① 顶部阶段标题行；② 横跨整宽的轴线（水平 line + 各阶段圆点节点）；③ 下方等宽卡片列。这是最稳定的时间线实现。

### 布局填满且不破
`body { overflow: hidden; }`；根容器用 `display: flex; flex-direction: column; height: 540pt;`，正文区 `flex: 1; min-height: 0; overflow: hidden;` 并让网格/表格 `height: 100%;` 拉伸——既从结构上杜绝下半页留白，又保证估错时只在容器内裁切、不撑破画布。

## 单页质量要点（写时一次写对，不回头返工）

### A. 版面与密度
1. 正文是否从标题分隔线排到了脚注线，底部空白 ≤ 40pt？
2. 纯文字页正文字数是否 ≥ 300（目标 450–800）？图文混合页解读文字是否 ≥ 150？
3. 每张卡片/每个小节是否 ≥ 4 行、每个表格是否 ≥ 7 行？

### B. 信息表达（新增，重点）
4. **本页是否包含 ≥ 1 个非纯文字的视觉表达**（对比图/构成图/趋势图/流程图/架构图/热力矩阵/评分标尺/雷达/时间线/决策矩阵）？若本页核心是数值对比/构成/趋势/多维/流程且全页只有文字和表格 → 不合格，必须补可视化。
5. **本页的图表类型是否与上一页不同？** 若上一页用了水平对比柱，本页不得再用水平对比柱——必须换雷达/堆叠条/时间线/热力矩阵/决策矩阵/点图等其他类型。
6. **本页图表类型是否匹配该页的数据结构？** 多维综合对比→雷达/热力矩阵（不要退化为多组条形）；构成占比→堆叠条；时间演进→时间线；决策建议→决策矩阵。参考「图表类型多样性」匹配表。
7. 本页的文字、表格、图表或图示是否确实最有利于当前论证，而不是沿用固定模板或为了可视化而可视化？
8. 若本页有对比图，对比主体的色系是否与全 deck 定义一致？
9. **全 deck 的彩色（非中性灰）是否 ≤ 3 个（主色 + 1 个对比主体色 + 强调红）？** 若热力矩阵/构成条引入了额外的彩色档（如同时出现蓝系 + 紫系档位）→ 配色混乱，不合格。
10. **对比主体 B 色是否与主色蓝有足够色相距离？** 若 B 色是紫色系（与蓝太近）→ 区分度不足，不合格，须改为红/琥珀/墨绿等暖色或对比色。

### C. 版式与图表多样性（全 deck 层面，写时留意）
11. 本页主版式是否与上一页相同？若相同 → 需换版式或合并页。
12. （写时留意全 deck 多样性）整套 deck 是否覆盖 ≥ 3 种不同主版式？
13. **（写时留意全 deck 多样性）整套 deck 是否出现了 ≥ 3 种不同图表类型？** 若全部图表都是水平对比柱 = 视觉单调。

### D. 逻辑与证据
14. 每条要点是否都是完整句子并带具体证据（数字/日期/名称/原话）？
15. 页内各小节是否复用了同一组加粗标签维度？
16. 是否有「本页结论」横条或等价的判断性收底？
17. **对比表/指标卡里是否存在双方数据缺失或完全相同的无效行？若有 → 删除或替换。**

### E. 技术约束
18. 是否无任何元素溢出 960×540pt？垂直预算合计是否 ≤ 正文区高度、文本框底边距画布底部是否 ≥ 0.5in？
19. 图表是否遵守可编辑 PPTX 约束（纯色 div 表达长度、不用 gradient/conic-gradient、文本不直接写进背景 div）？
20. **本页的连接线/竖线是否用了固定 height，而非垂直 flex:1 拉伸？** 若用了 flex:1 拉伸竖线 → 高度不确定时会塌陷，不合格。
21. **本页的多区域布局（如四象限、多列卡片）是否用了 CSS grid 或横向 flex 等分，而非绝对定位？** 若用绝对定位画多象限 → 容易重叠，不合格。

### F. 字号与可读性（新增，硬约束）
22. **本页所有可见文字的 font-size 是否 ≥ 10px？** 逐个检查：正文、卡片正文、表格单元格、图表数值标签、图例、轴标签、时间线卡片、堆叠条内/外文字、热力矩阵单元格、脚注。任一 < 10px → 不合格，必须抬到 ≥ 10px（正文 ≥ 12px）。
23. **正文（卡片正文/段落/解读区）是否 ≥ 12px？** 若正文是 10-11px → 偏小，不合格，必须抬到 12-14px。
24. **当内容装不下时，是否优先拆页/减模块，而非缩字号硬塞？** 若为了塞下内容把字号压到下限边缘 → 不合格，应拆成两页或减少次要模块。

## 避免

### 信息表达层面（重点）
- **不把 raw 数据当信息堆砌**：纯数字表格 + 「数据来源」脚注不是分析，是数据附录。每个数字都必须被转成对比/构成/趋势的视觉表达 + so what 判断。
- **不用固定模板套所有页面**：连续多页左右双栏卡片 / 连续多页满版表格 = 版式同质化，不合格。
- **不只用一种图表类型**：8 页 deck 里 4 页都是水平对比柱 = 图表单调，不合格。多维对比要用雷达/热力矩阵，构成要用堆叠条，趋势要用时间线，决策要用决策矩阵——按数据结构选图。
- **不把最有张力的数据埋进表格**：像「1196 vs 48」这种数量级鸿沟，必须提取成独立的大对比图，不能埋在 7 行表格的第一行里。
- **不画无信息量的图**：图表必须服务于标题结论。为「看起来有数据」而硬加的图，和为填满页面而硬堆的字，一样是 slop。
- **不用无效数据凑行**：对比双方都「未公开」或数值相同的指标，没有对比张力，必须删除或替换。

### 文字层面
- **不留下半页空白**——这是本风格最严重的失败模式之一；宁可多画一张图、多写一组维度，也不留白。
- **不写关键词式短 bullet 页**：只有名词短语的 3–5 条大字要点页是演讲风格，视为不合格输出。
- 不写两行小卡片拼贴页：卡片必须是多字段实体，不是标签贴纸；结论页的决策建议不得用小 chip 拼贴。
- 不做大留白页、巨型 hero 数字页、情绪化金句页；不为「呼吸感」牺牲信息量。
- 不写没有结论的罗列：标题必须给出判断，正文与图表必须支撑标题。
- 不把长内容压成不可读：宁可拆页，**不许任何可见文字字号低于 10px**（正文 ≥ 12px）、不许行距挤压、不许溢出画布。**绝不用缩字号来硬塞内容**——这是最常见也最致命的可读性失败，宁可少写一行、拆成两页，也不许把 11px 正文压成 9px。
- 不省略证据：分析必须落到具体的数字、文件、函数、原话或行为，不许停留在「效果不错」「存在问题」这类空泛判断。

### 视觉层面
- 不使用装饰性图形、图标墙、大面积色块和渐变；所有视觉元素必须承载信息。
- 不随意更换对比主体色系：一旦在封面定义，全 deck 必须稳定复用。
- **不让全 deck 彩色超过 3 个**（主色 + 1 对比主体色 + 强调红）。热力矩阵/构成条的深浅档必须用主体色的同色系明度变化，不得引入新的彩色——蓝系 + 紫系同时出现 = 配色混乱。
- **不选紫色系作为对比主体 B 色**：紫色与主色蓝色相太近，放一起像「深蓝 vs 蓝紫」区分度不够。B 色须选红/琥珀/墨绿等暖色或对比色。
- **不用垂直 flex:1 拉伸连接线/竖线**：高度不确定时竖线会塌陷。连接线一律用固定 height 的 div。
- **不用绝对定位画多象限/多区域布局**：只设 top 不设 height 会无限撑开导致重叠。多区域等分一律用 CSS grid。

## 输出要求
保留输入内容的信息结构和文字含义，按此风格重绘为统一连贯的 16:9 演示页面。**核心动作是把 raw 数据转化为有效信息的视觉表达**：输入中的数字对比要用对比图呈现，构成要用构成图呈现，多维综合要用雷达/热力矩阵，时间演进要用时间线，决策建议要用决策矩阵，流程要用流程图，机制论证用完整句子的详尽文字展开。**图表类型必须多样化，不得全 deck 只用水平对比柱一种**。输入内容单薄时必须基于输入展开成完整句子的详尽分析并补齐分析维度，直到满足画布填充硬指标，而不是缩减成短语或留白。所有视觉元素（含表格、对比图、构成图、雷达近似、热力矩阵、时间线、决策矩阵、流程图、架构图、指标面板、点图）必须用 HTML/CSS 实现，不依赖外部图片资源。
